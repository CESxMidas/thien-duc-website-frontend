1. Tóm tắt Phương án 2

Phương án 2 tách hệ thống thành 3 phần riêng biệt:

| Phần | Công nghệ | Domain (đề xuất) | Vai trò |
|------|-----------|------------------|---------|
| Website công khai | Next.js 16 + React 19 | `thienduc.vn` | Khách xem nội dung, gửi form liên hệ |
| Trang quản trị | Vite + React + TypeScript | `admin.thienduc.vn` | Nhân viên cập nhật nội dung, xử lý liên hệ |
| Backend API | NestJS + Prisma | `api.thienduc.vn` | Lưu trữ, phân quyền, xử lý nghiệp vụ |

Điểm khác biệt so với Phương án 1:
- Không gộp admin vào chung một project với website.
- Admin sửa nội dung không cần dev sửa code.
- Form liên hệ có trạng thái xử lý và lưu database.
- Tài khoản có phân quyền cơ bản (3 cấp).

Điểm khác biệt so với Phương án 3:
- Chưa cần báo cáo phức tạp, staging đa tầng, sao lưu tự động nhiều lớp.
- Chưa cần phân quyền chi tiết từng trường dữ liệu (dành cho mở rộng sau).

Thời gian ước lượng: 8–10 tuần (website + API + admin + tích hợp + UAT + go-live).

Hiện trạng codebase: Frontend Next.js đã có ~70% giao diện với data tĩnh (`src/data/`). Backend, admin và song ngữ Vi/En chưa triển khai. Giai đoạn tiếp theo: hoàn thiện báo cáo → phê duyệt → triển khai API/admin, không mở rộng data tĩnh tùy tiện.

2. Đối tượng sử dụng

Website Thiên Đức là trang giới thiệu doanh nghiệp, không phải sàn bán hàng. Hệ thống không quá phức tạp nhưng phải trả lời được:

- Ai cập nhật nội dung khi có dự án mới?
- Form liên hệ ai phản hồi, có lưu lại không?
- Phân quyền cập nhật nội dung thế nào?

2.1 Khách truy cập

- Vào website xem thông tin công ty, dự án, tin tức.
- Gửi yêu cầu liên hệ / tư vấn qua form.
- Không cần đăng nhập, không có tài khoản khách.

2.2 Nhân viên bộ phận nội dung (Editor)

- Soạn nội dung dự án, tin tức, banner theo quyền được cấp.
- Upload ảnh lên Cloudinary.
- Lưu bài ở trạng thái nháp hoặc gửi duyệt — không publish trực tiếp.

2.3 Admin

- Duyệt và publish nội dung.
- Xem và xử lý form liên hệ (đổi trạng thái, ghi chú nội bộ).
- Quản lý banner, dự án, tin tức, trang tĩnh.

2.4 Super Admin

- Toàn quyền: tạo/xóa tài khoản nhân viên.
- Cập nhật thông tin quan trọng công ty (phone, email, địa chỉ, metadata).
- Giai đoạn đầu: có thể chỉ cần 1 tài khoản Super Admin; mở rộng sau.

2.5 Phân quyền PA2

Chỉ cần 3 cấp: `SUPER_ADMIN`, `ADMIN`, `EDITOR`.  
Chưa cần phân quyền từng trường dữ liệu (Phương án 3).

3. Quy trình nghiệp vụ

3.1 Form liên hệ (khách → admin)

```
Khách điền form trên /lien-he
        ↓
Website POST /public/contacts (HTTPS)
        ↓
API validate + chống spam (rate limit)
        ↓
Lưu contact_submissions — trạng thái NEW
        ↓
Gửi email thông báo cho admin (Gmail)
        ↓
Hiển thị "Gửi thành công" trên website
        ↓
Admin đăng nhập → xem chi tiết → IN_PROGRESS → DONE
```

Lưu ý: Admin không gửi email trả khách tự động trong hệ thống; trả lời bằng phone/email thủ công, trên admin chỉ cập nhật trạng thái và ghi chú.

[HÌNH ẢNH] `docs/diagrams/02-form-lien-he.drawio` — export PNG, sửa lane Admin = Vite + React.

----------------------------------------

3.2 Tin tức (Editor → Admin)

```
Editor viết bài + upload ảnh cover (Cloudinary)
        ↓
Lưu DRAFT hoặc gửi PENDING_REVIEW
        ↓
Admin duyệt → PUBLISHED
        ↓
Website hiển thị bài (ISR revalidate)
```

----------------------------------------

3.3 Dự án

```
Admin thêm/sửa dự án: tên, slug, trạng thái, mô tả, gallery, bản đồ
        ↓
Thêm hạng mục con (project_items) nếu có
        ↓
Publish
        ↓
/du-an lọc theo trạng thái; /du-an/[slug] hiển thị chi tiết
        ↓
/du-an/[slug]/[hang-muc] hiển thị hạng mục (bổ sung theo yêu cầu nghiệp vụ)
```

Trạng thái dự án: Đã bàn giao | Đang thi công | Chuẩn bị khởi công.

----------------------------------------

3.4 Banner trang chủ

```
Admin sửa nội dung + ảnh banner
        ↓
Lưu qua API
        ↓
Website cập nhật ở lần tải tiếp theo (ISR 60s–300s)
```

----------------------------------------

3.5 Tạo tài khoản nhân viên

```
Super Admin tạo user (email, role)
        ↓
Gửi thông tin đăng nhập cho Editor/Admin
        ↓
Nhân viên đăng nhập admin.thienduc.vn
```

4. Tổng thể hệ thống

```
┌─────────────────┐     HTTPS      ┌─────────────────┐
│  thienduc.vn    │ ──────────────►│  NestJS API     │
│  (Next.js)      │   public API   │  Render/Railway │
└─────────────────┘                └────────┬────────┘
                                            │
┌─────────────────┐     HTTPS JWT           │
│ admin.thienduc  │ ────────────────────────┤
│ (Vite+React)    │   admin API             │
└─────────────────┘                         │
                                     ┌──────▼──────┐
                                     │ PostgreSQL  │
                                     │  (Prisma)   │
                                     └─────────────┘
                                            │
                                     ┌──────▼──────┐
                                     │ Cloudinary  │
                                     │    CDN      │
                                     └─────────────┘
```

Nguyên tắc:
- Website và admin dùng URL ảnh từ Cloudinary, không lưu file nặng trên server API.
- Website public chỉ gọi endpoint public/; admin gọi endpoint có JWT.
- Website deploy Vercel (tải nhanh, SEO); admin tách riêng không ảnh hưởng SEO.

5. Domain và môi trường

| Môi trường | Website | Admin | API | Database |
|------------|---------|-------|-----|----------|
| Production | thienduc.vn | admin.thienduc.vn | api.thienduc.vn | PostgreSQL prod |
| Staging | staging.thienduc.vn | admin-staging.thienduc.vn | api-staging.* | PostgreSQL staging (schema riêng) |

Ghi chú: PA2 không cần hạ tầng staging phức tạp như PA3, nhưng bắt buộc có staging để test trước production. Có thể dùng Render/Neon với schema PostgreSQL riêng.

| Dịch vụ | Mục đích |
|---------|----------|
| Vercel | Host Next.js + static admin |
| Render / Railway | Host NestJS + PostgreSQL |
| Cloudinary | CDN ảnh/video |
| Gmail / Google Workspace | SMTP thông báo form |

Chi phí domain: tham chiếu `docs/so-sanh-domain-trong-nuoc-va-vercel-moi.xlsx`.

6. Thiết kế cơ sở dữ liệu

(Bổ sung phiên bản 2.0 — thay cho ghi chú “chưa thiết kế” trong bản cũ.)

6.1 Bảng chính

| Bảng | Mô tả |
|------|--------|
| `users` | Tài khoản admin (email, hash, role) |
| `projects` | Dự án (slug, title_vi/en, status, …) |
| `project_items` | Hạng mục trong dự án |
| `project_gallery_images` | Ảnh gallery (url, public_id, sort) |
| `news_posts` | Tin tức (slug, status, published_at) |
| `news_categories` | Danh mục tin (tuỳ chọn) |
| `pages` | Trang tĩnh (HR, giới thiệu mở rộng) |
| `banners` | Banner trang chủ |
| `contact_submissions` | Phiếu liên hệ từ form |
| `site_settings` | Cài đặt key-value (phone, email, …) |

6.2 Quan hệ chính

- `projects` 1—N `project_items`
- `projects` 1—N `project_gallery_images`
- `project_items` 1—N `project_gallery_images`
- `users` 1—N `news_posts` (author)
- `contact_submissions` N—1 `inquiry_types` (enum hoặc bảng)

6.3 Song ngữ (Vi/En)

Các trường nội dung chính có hậu tố `_vi` và `_en`:
- `title_vi`, `title_en`
- `summary_vi`, `summary_en`
- `content_vi`, `content_en` (JSON hoặc text)

Website public truyền `?locale=vi|en` hoặc prefix `/vi/`, `/en/`.

7. Backend API (NestJS)

7.1 Modules

| Module | Chức năng |
|--------|-----------|
| `auth` | Login, JWT, refresh (tuỳ chọn) |
| `users` | CRUD user (Super Admin) |
| `site-settings` | Phone, email, địa chỉ, metadata |
| `banners` | CRUD banner |
| `projects` | CRUD dự án + hạng mục + gallery |
| `news` | CRUD tin, workflow duyệt |
| `pages` | CRUD trang tĩnh |
| `contacts` | Public submit + admin quản lý |
| `media` | Cloudinary upload signature |

7.2 Endpoint public (website)

| Method | Path | Mô tả |
|--------|------|--------|
| GET | `/public/projects` | Danh sách dự án |
| GET | `/public/projects/:slug` | Chi tiết dự án |
| GET | `/public/projects/:slug/items/:itemSlug` | Chi tiết hạng mục |
| GET | `/public/news` | Danh sách tin |
| GET | `/public/news/:slug` | Chi tiết tin |
| GET | `/public/pages/:slug` | Trang tĩnh |
| GET | `/public/banners` | Banner active |
| GET | `/public/settings/contact` | Thông tin liên hệ |
| POST | `/public/contacts` | Gửi form |

7.3 Endpoint admin (JWT)

| Method | Path | Mô tả |
|--------|------|--------|
| POST | `/auth/login` | Đăng nhập |
| GET | `/admin/contacts` | Danh sách liên hệ |
| PATCH | `/admin/contacts/:id` | Cập nhật trạng thái / ghi chú |
| GET/POST/PATCH | `/admin/projects` | CRUD dự án |
| GET/POST/PATCH | `/admin/news` | CRUD tin |
| GET/PATCH | `/admin/pages/:slug` | Trang tĩnh |
| GET/PATCH | `/admin/banners` | Banner |
| POST | `/admin/media/upload-signature` | Ký upload Cloudinary |
| GET/PATCH | `/admin/settings` | Cài đặt site |
| GET/POST/PATCH | `/admin/users` | User (Super Admin) |

7.4 Bảo mật đăng nhập

| Quy tắc | Giá trị |
|---------|---------|
| Mật khẩu tối thiểu | 10 ký tự, có chữ và số |
| Khóa tài khoản | Sau 5 lần sai → 15 phút (có thể tăng dần) |
| Xác thực email | Bắt buộc khi tạo tài khoản admin |
| Request admin | Header `Authorization: Bearer <token>` |

7.5 Form liên hệ — hiện trạng vs mục tiêu

| | Hiện tại | Sau PA2 |
|---|----------|---------|
| Gửi form | `mailto:` trên máy khách | POST API |
| Lưu trữ | Không | PostgreSQL |
| Thông báo | Không | Email admin |
| Quản lý | Không | Trang admin |

7.6 Upload ảnh

```
Admin chọn file
        ↓
Admin gọi POST /admin/media/upload-signature
        ↓
API trả signature Cloudinary
        ↓
Admin upload trực tiếp lên Cloudinary
        ↓
Lưu URL + public_id khi save dự án/tin/banner
```

10. Phân quyền

| Vai trò | Nội dung | Liên hệ | Hệ thống |
|---------|----------|---------|----------|
| EDITOR | Tạo/sửa nháp, không publish | Không | Không |

| SUPER_ADMIN | Đầy đủ | Đầy đủ | Cài đặt + user |

Trạng thái tin tức

| Trạng thái | Hiển thị web |
|------------|--------------|
| DRAFT | Không |
| PENDING_REVIEW | Không |
| PUBLISHED | Có |

Trạng thái liên hệ

| Trạng thái | Nhãn |
|------------|------|
| NEW | Mới |
| IN_PROGRESS | Đang xử lý |
| DONE | Đã xong |
| SPAM | Spam |
