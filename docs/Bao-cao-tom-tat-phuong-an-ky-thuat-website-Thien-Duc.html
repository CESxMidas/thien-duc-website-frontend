1. MỤC ĐÍCH

Trình bày 3 phương án kỹ thuật. Mỗi PA: so sánh ưu/nhược từng công nghệ → lý do chọn → đánh giá mở rộng, bảo trì, tốc độ phản hồi.

  • Hiện trạng: Frontend Next.js 16 đang phát triển, mock data trong src/data/.
  • Yêu cầu: Website + Backend + CSDL + Admin + form liên hệ + tuyển dụng.

3. PA1 — Static-first (SSG / Jamstack)

Ưu tiên: TỐC ĐỘ RA MẮT GIAO DIỆN và CHI PHÍ THẤP NHẤT

Website dạng trang tĩnh. Nội dung trong src/data/. Không backend, không CSDL, không admin. Phù hợp giai đoạn 1 — demo giao diện.

3.1. So sánh công nghệ và lý do chọn

Frontend — tại sao chọn Next.js?

Công nghệƯu điểmNhược điểm
Next.js 16 ✓SSG/ISR; SEO tốt; đang dùng — không đập lại FEApp Router cần học
AstroTrang tĩnh rất nhanhPhải migrate toàn bộ FE
RemixForm/action tốtPhải viết lại; ít tài liệu VN
React SPA (Vite)Dev nhanhSEO kém — không phù hợp web giới thiệu

→ Lý do chọn:

Chọn Next.js 16 vì: (1) dự án đã code sẵn Next.js; (2) SSG nhanh, SEO tốt; (3) sau này chỉ thay layer data, không đổi UI.

Dữ liệu — tại sao KHÔNG dùng CSDL?

Phương ánƯu điểmNhược điểm

→ Lý do chọn:

Chọn file trong repo vì PA1 ưu tiên đơn giản, chi phí 0. Khi cần CSDL → chuyển PA3.

PA1 KHÔNG có CSDL — chủ đích, không phải thiếu sót.

4. PA2 — Headless CMS

Ưu tiên: TỰ QUẢN TRỊ NỘI DUNG — marketing tự đăng bài

Next.js + CMS (Strapi/Sanity). CMS có CSDL riêng. Form/HR vẫn thiếu — dễ thành 2–3 hệ thống rời.

4.1. So sánh công nghệ và lý do chọn

Frontend — Next.js

Công nghệƯu điểmNhược điểm
Next.js 16 ✓ISR từ CMS; tái dùng FE hiện tạiCần cấu hình revalidate
Nuxt.jsTương tự cho VueTeam đang dùng React

→ Lý do chọn:

Chọn Next.js vì tái dùng FE đang code; ISR giữ trang nhanh khi có bài mới.

CMS — chọn Strapi 5

CMSƯu điểmNhược điểm

SanityCloud; preview tốtVendor lock-in; chi phí scale
ContentfulEnterpriseĐắt khi nhiều user

→ Lý do chọn:

Chọn Strapi vì open-source, tự host, dùng PostgreSQL — đồng bộ với PA3.

Strapi là CMS + backend nội dung — KHÔNG xử lý HR/form phức tạp.

CSDL — PostgreSQL (qua Strapi)

CSDLƯu điểmNhược điểm
PostgreSQL ✓ACID; Strapi hỗ trợ nativeCần server DB
Sanity cloudManagedVendor lock-in

→ Lý do chọn:

PA2 CÓ CSDL — nhưng chỉ cho nội dung (tin, dự án). Form/CV không nằm đây.

Backend nghiệp vụ — CHƯA ĐỦ

Chỉ CMS ✓Đơn giản cho contentKhông form/CV/HR

→ Lý do chọn:

PA2 thiếu backend nghiệp vụ — lý do không đề xuất cho Thiên Đức.

5. PA3 — Full-stack (FE + BE + CSDL + Admin)

Ưu tiên: KIỂM SOÁT TOÀN DIỆN, MỞ RỘNG, MỘT HỆ THỐNG THỐNG NHẤT

3 repo: Frontend Next.js + Backend NestJS + PostgreSQL + Admin Vite+React. Một API cho cả web và admin.

5.1. So sánh công nghệ và lý do chọn

Frontend — Next.js

Công nghệƯu điểmNhược điểm
Next.js 16 ✓SSG/ISR từ API; tái dùng FEPhức tạp hơn SPA
React SPAĐơn giảnSEO kém

→ Lý do chọn:

Chọn Next.js vì tái dùng 100% giao diện; publish admin → revalidate trang public.

Backend — NestJS (KHÔNG phải CSDL)

FrameworkƯu điểmNhược điểm
NestJS 11 ✓TypeScript; module rõ; Swagger; RBACBoilerplate nhiều
Express.jsNhẹDễ lộn xộn khi scale

→ Lý do chọn:

Chọn NestJS vì cùng TypeScript với FE/Admin; cấu trúc module dễ bảo trì.

⚠️ NestJS chỉ là framework xử lý logic — CẦN CSDL bên dưới.

CSDL — PostgreSQL (BẮT BUỘC)

CSDLƯu điểmNhược điểm
PostgreSQL 16 ✓ACID; quan hệ; full-text search; Neon/SupabaseCần thiết kế schema

→ Lý do chọn:

Chọn PostgreSQL vì dữ liệu có quan hệ; form/CV cần ACID.

Sơ đồ: NestJS → Prisma → PostgreSQL.
